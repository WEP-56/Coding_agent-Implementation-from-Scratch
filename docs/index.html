Reality Check

这个仓库现在到底有什么

Python Agent Core

  • 统一的 contracts：Task、Plan、PatchSet、ToolCall、ToolResult、Event
  • Repo workspace sandbox，限制路径逃逸与直接越界读写
  • SQLite 持久化：run、step、tool_call、event
  • Manifest / symbol / repo_map 索引链路
  • 统一 diff patch 工具与 CLI 命令
  • Telegram 本地模拟入口

Desktop Prototype

  • React + TypeScript 三栏工作台
  • Tauri Rust 命令桥接本地仓库读写与 patch 应用
  • Plan / Build / Auto 模式与审批队列
  • Artifacts、Rollback、Trace 导出、Memory Blocks
  • OpenAI-compatible tool loop 的桌面端实验实现

仍然缺失的关键闭环

  • Python core 与桌面端并没有共享同一个权威运行时
  • 上下文压缩、恢复、fork、replay 仍未产品化
  • 索引当前以 Python AST 为主，多语言能力仍很浅
  • 验证、观测、安全策略还没有统一成一条工程流水线

GitHub Pages

这套文档如何发布

为什么现在就适合上 Pages

文档目录采用纯静态 HTML + CSS，发布时不需要额外的 Node、Python 或 Jekyll 构建步骤。 这对一个还在频繁调整架构的原型项目很重要，因为文档发布链越简单，越不容易在后来失真。

最小发布步骤

  1. 把仓库推到 GitHub。
  2. 在仓库设置中打开 GitHub Pages。
  3. Source 选择当前分支的 /docs 目录。
  4. Pages 生效后，以 docs/index.html 作为首页。
Documentation Policy

从现在开始，哪些文档算“当前事实”

当前事实来源

  • docs/index.html 到 docs/roadmap.html
  • 根目录新的 README.md
  • docs/roadmap/ 内仍有价值的内部规格

历史草稿的定位

根目录保留的早期规划文档现在应视为“设计过程留痕”，而不是要继续直接暴露给最终用户的说明书。 它们对研发回溯仍有参考价值，但后续若与新文档冲突，应以这里的文档站点为准。
