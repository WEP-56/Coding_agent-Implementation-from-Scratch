不抄外壳，系统性模仿 Codex 的运行时语法和反馈闭环

这份文档的核心判断是：CodingGirl 当前最大问题不是“少几个功能”，而是 agent runtime、context lifecycle、tool / diff provenance、以及面向用户的反馈语法都还没有形成闭环。 后续改造要保留我们“单工作流卡片承载动态 / Diff / Tools”的独特设计，但其下层执行系统必须向 Codex / OpenCode 的成熟做法靠拢。

Diagnosis

为什么 Codex / OpenCode 用起来比我们好

1. 它们有明确的 canonical session grammar Turn、tool、command、diff、approval、compaction、rollback 都是一等对象，而不是后端一个状态、前端再拼一个状态。
2. 它们的“模式”是真正的 agent 语义，不是 UI label Build / Plan / Explore / Review 会直接影响工具权限、执行循环和上下文策略，而不是只切一个按钮颜色。
3. 它们在 repo anchoring 上更强 用户进入某个项目后，模型默认就站在那个工作区里思考，不容易出现“像没进 repo”一样的错位。
4. 它们的首帧反馈更快 任务开始后会先出现稳定的 run / turn shell，再不断填充 reasoning、tools、diff，而不是让用户等轮询补数据。
5. 它们把 terminal / diff / review 做成长期状态 OpenCode 的 terminal 是 workspace-scoped 且可序列化恢复；Codex 则把 diff、review、worktree、skills 融进一个长期工作流。
6. 它们的失败恢复语义更稳定 provider 抖动、迟到事件、context compaction、rollback、approval 都有更清晰的系统级 fallback，而不是某一层失败就整轮体验断掉。
7. 它们更重视测试和可观测性 Codex 上游大量测试 turn reconstruction；OpenCode 有 terminal / prompt / keybind / session history 的 e2e，说明体验层是被工程化约束的。
Design Position

我们要模仿什么，不模仿什么

要模仿的

  • Turn / Item / Event / Context 的统一语义
  • Mode 与 permission 的强绑定
  • Diff provenance、approval、rollback 的闭环
  • workspace-scoped terminal、history、session continuity
  • compaction、summary、resume/fork 的产品化上下文生命周期
  • 系统级 fallback、测试和观测能力

不应该抄的

  • 单纯照搬 Codex 的界面外观
  • 为了“像 Codex”而放弃我们的工作流卡片设计
  • 先做云端并行、多 agent 编排，再回头补基础运行时
  • 继续靠功能堆叠掩盖 runtime 不稳定
Gap Table

P0-P3 缺口与改造重点

优先级 缺口 为什么重要 目标状态
P0 Canonical TurnItem model 解决 runtime、timeline、workflow card、diff、approval 各记一份状态导致的漂移问题。 所有执行事实先落成 turn items，再投影成 timeline / cards / exports。
P0 流式事件总线替代轮询 直接影响“卡片是否秒出”“用户是否觉得慢”。 发送消息后立即出现 run shell，后续状态以 push 方式增量更新。
P0 Repo anchoring / session preflight 解决“会话明明关联仓库，但模型像没站在 repo 里工作”的严重错位。 每轮任务前明确注入 workspace identity、repo facts、session intent。
P0 Universal diff provenance 所有写操作都必须可 review、可 rollback、可归属 run。 patch、direct write、approval replay、manual edit 都统一产出 diff / artifact。
P0 Failure taxonomy + retry semantics provider 抖动、transport error、tool error、approval block 不能再全部表现成“执行失败”。 错误分类清晰，可恢复路径明确，可局部重试。
P1 Context Engine v2 当前只有 history/summary/memory 的轻量拼装，离 Codex-grade context lifecycle 还差一代。 有 normalization、compaction、prune tool outputs、token breakdown、context debug。
P1 Model-first intent router 自然语言任务分流不能永远靠硬编码规则。 模型优先、规则兜底、路由原因可视化、失败可回退。
P1 Agent / subagent system 仅有 plan/build/auto 不足以支撑复杂探索、review、compaction、general search。 Build / Plan / Explore / Review / General 明确化，且有 hidden internal agents。
P1 Terminal v2 终端是 coding product 的核心，不是附属工具。 workspace-scoped、可持久化、多 tab、与 session/run 联动。
P2 Resume / Fork / Rollback semantics 让 session 真正变成可恢复、可分叉、可审查的长期对象。 fork 不丢上下文，rollback 不只是“把文件写回去”，而是 session event graph 的一部分。
P2 Validation / review pipeline 从“会改代码”提升到“会稳定交付工程结果”。 lint/test/build/review comments 都成为统一 item 和反馈层的一部分。
P2 Watcher / workspace awareness 减少无关文件噪音，提升模型对真实工作区的感知。 watch ignore、workspace cache、context invalidation 规则化。
P3 Skills / reusable workflows 提升长期效率，但不应先于基础 runtime 闭环。 团队级可复用技能、项目级 recipes、可共享配置。
P3 Cross-client continuity 长期会重要，但不是当前可用性的第一阻塞。 desktop / CLI / future IDE 共享 session、history、config、skills。

P0: 先把系统跑稳

  1. 统一 TurnItem schema，并用它驱动 timeline / run card / trace export。当前已把 phase / context / model_request / validation 接入 SessionTurnItem。
  2. 把聊天区与右侧栏的数据源改成事件流，不再依赖轮询拼状态。
  3. 发送消息后立即创建 optimistic run shell，再流式填充动态；workflow card 主视图优先读取 SessionTurn.items，timeline 退回 projection。
  4. 做 session preflight，确保每轮都有明确 repo identity 和 workspace anchoring。
  5. 所有 mutation 统一产出 diff / artifact / rollback metadata / runId。
  6. 拆出 transport / model / tool / approval / validation 的错误分类与重试规则。

P1: 再把系统跑聪明

  1. 把 intent router 升级成模型判定优先、规则兜底、路由原因可观测。
  2. 做 Context Engine v2：normalization、compaction lifecycle、tool-output prune、token breakdown。
  3. 把 Build / Plan / Explore / Review / General 做成真正的 agent 语义。
  4. 把 terminal 升级成 workspace-scoped、多 tab、可序列化恢复。
  5. 把 context debug / session context 面板做成长期维护能力，而不是一次性调试面板。

P2: 让它接近产品级

  1. 补齐 resume / fork / rollback 的统一 session semantics。
  2. 把 test/build/lint/review 做成 canonical turn items 和反馈闭环。
  3. 引入 workspace watcher、ignore patterns、context invalidation。
  4. 做长期任务 supervision，而不是每轮只看一次模型回复。

P3: 最后补长期效率层

  1. 引入 skills / reusable workflows / project recipes。
  2. 评估 CLI / desktop / future IDE 的 shared session continuity。
  3. 补 provider-native adapters、team config、长期 automation。
Decision Rule

后续所有设计决策的判断标准

这个能力有没有 canonical object？ 没有一等对象，就不要先做 UI。
它和上下游有没有闭环？ 能否从 runtime 一直追到 diff、approval、artifact、rollback、front-end feedback。
用户能否明确感知当前状态？ 如果系统在工作，但用户不知道它在干什么，就是设计失败。
失败后能否局部恢复？ 不能局部恢复，就说明系统语义还不够强。
