Principles

真正决定 coding agent 成败的，不只是模型，而是结构、权限和可恢复性

如果只看表面，CodingGirl 像是在“同时做一个 Python agent core 和一个桌面端 AI IDE”。 但从工程角度看，更准确的说法是：它正在摸索一个统一的 coding agent 运行时，只是目前还没把所有层收拢到同一条执行链上。

关键词
Diff First Trace First Repo Map Approval Gate
当前最大裂缝

Python core 和桌面端都在做编排，但还没有共享同一个权威状态机、事件协议和恢复语义。

Design Goals

这个项目真正应该坚持的设计原则

模型无关

内部用统一 contracts 与 tool schemas，把模型 provider 适配放到边界层，而不是把业务逻辑绑死在某个 API 上。

先读后写

先通过 repo map、search、read_file 等只读工具定位上下文，再尝试 patch，避免“直接改错地方”的高成本失败。

可追踪可恢复

任何运行都应该能回答：调用了什么工具、做了什么改动、为什么失败、怎样回滚或重放。

Python Core

Python 这条线已经沉淀了什么

1. Contracts 是基础语言

codinggirl/core/contracts.py 定义了 Task、Plan、PatchSet、ToolCall、ToolResult、Event， 这是整个系统“模型无关”的基石。只要这些对象是稳定、可序列化、可落盘的，后面再替换 LLM provider、 再替换界面层，都不会让运行时完全失控。

2. Workspace Sandbox 定义访问边界

RepoWorkspace 负责拒绝绝对路径、阻止 .. 逃逸、限制读取大小，并统一文本写入方式。 这一步并不花哨，却是从“脚本玩具”迈向“工程工具”的前提。

3. Tool Runner + SQLite Trace

Python 工具调用会记录到 SQLite 的 tool_call 与 event 表里。哪怕今天它还没有完整 replay 模式，这种结构也已经为后续的恢复、调试和审计留出了位置。

4. Repo Map 不是向量库替代品，而是最小理解层

当前索引链路由 manifest.json、symbols.sqlite 和 repo_map.txt 组成。 它先覆盖 Python AST 级别的 class / function / import，再根据重要性和焦点词生成适合喂给模型的仓库摘要。

Patch Strategy

为什么要坚持 diff-first，而不是整文件重写

当前做法

Python 侧通过 patch_apply_unified_diff 支持 fail-on-conflict 的统一 diff 应用， 会先解析 patch，再按 hunk 顺序应用，并在冲突时拒绝写入。对于桌面端，则另外实现了更接近 Codex 风格的 patch 应用和 artifact 持久化。

这样做的意义

  • 变更范围更小，更容易被审查
  • 失败原因更明确，例如 context mismatch、delete mismatch
  • 更容易做 rollback 与 artifact 保存
  • 后续更容易和审批、trace、PR 生成衔接
Desktop Runtime

桌面端原型为什么不是“纯 UI”

前端层做了什么

React 前端不只是展示对话，它维护仓库、会话、时间线、Diff、Tool Calls、Logs、Artifacts、Memory Blocks 等状态，并通过桥接层把数据请求转给 Tauri 命令。

Rust 命令层做了什么

Tauri 的 Rust 命令层负责读取仓库树、读写文件、应用 patch、处理审批请求、导出 trace bundle、记录本地会话状态。 这意味着桌面端其实已经拥有一个“本地 agent shell”的雏形。

桌面端的模型工具循环

apps/desktop/src-tauri/src/commands/llm.rs 实现了一套 OpenAI-compatible 的 tool loop： 它会构造 system prompt、附加 repo context 与 memory blocks、接收 tool calls、根据模式与策略进行审批， 再把工具结果送回模型。它甚至已经开始尝试“read-before-write guard”这种产品级防线。

问题不在于这套东西不存在，而在于它目前并没有和 Python core 共用同一份 contracts、同一套状态恢复逻辑和同一条事件流水线。

Approval Model

审批、权限与 trace 为什么是产品核心

Plan / Build / Auto

模式不是 UI 装饰，而是 mutating tools 的执行门槛。Plan 禁写，Build 默认要审批，Auto 才能自动应用。

Ask / Allow / Deny

策略决定工具可否执行，并可按仓库保存。桌面端还实现了 session 级 permission cache。

Trace / Artifact / Rollback

没有这些就无法建立用户信任。用户需要知道系统做了什么，也需要能在出错时回头。

Gap Analysis

当前架构最需要正视的裂缝

裂缝 1：双运行时并存 Python core 和桌面端都在负责 orchestration，但它们没有共享同一个 session lifecycle。
裂缝 2：事件模型还不是真正的单一事实源 Python 有 SQLite event log，桌面端有 timeline / trace 结构，但二者没有统一协议。
裂缝 3：索引能力未跨语言统一 Python 侧现在主要覆盖 Python AST，桌面端也没有直接消费这套索引结果。
裂缝 4：恢复语义还不完整 rollback、artifact 和 trace export 已有雏形，但 resume / fork / replay 仍未形成产品闭环。
Why Codex / OpenCode Matter

从 Codex 和 OpenCode 应该学什么

该学 Codex 的地方

  • 把上下文管理、事件顺序、工具执行和恢复机制做成一等公民
  • 把 patch 应用、审查、trace 和批准流程织成完整生命周期
  • 把“像资深工程师那样工作”落实到 guardrails 与执行协议，而不是文案

该学 OpenCode 的地方

  • 清晰的权限模型
  • 多表面入口但共享会话语义
  • 规则、session、undo/redo、工程化可逆能力

不该学的地方

不要急着复制所有外观和入口。对于 CodingGirl 现在这个阶段，真正的优先级是“单一执行事实源、稳定工具协议、可恢复会话”， 而不是先把 UI 做得更像某个成熟产品。
