最大现实问题

“能跑”与“能交付”之间还差一整层产品级运行时设计，尤其是统一事件协议和可恢复会话。

What Is Real

当前已经真实落地的能力

基础设施

  • Workspace sandbox
  • SQLite trace / tool_call schema
  • Patch apply with conflict rejection
  • Manifest / symbols / repo_map

入口与适配器

  • CLI 命令集
  • Telegram 本地模拟
  • 桌面端 React + Tauri 原型

产品雏形

  • Mode 切换
  • 审批队列与 session permission
  • Artifacts、rollback、trace export
  • Memory blocks 与 chat summary
What Is Not Ready

当前还不能把它当成熟产品的原因

1. 没有单一执行权威 Python core 与桌面端原型都能做调度，但它们不是同一个 engine。
2. 当前 orchestrator 还很窄 Python orchestrator 只支持一种 replace goal 语法，不是完整 task planner。
3. 上下文生命周期仍不完整 压缩、恢复、fork、replay、session checkpoints 都没有形成稳定产品能力。
4. 多语言代码理解仍不足 索引以 Python AST 为主，还没有真正覆盖桌面端自身的 TypeScript / Rust 主工作流。
5. 工程发布链尚未打通 缺少统一版本管理、CI、发布流程、安装说明与用户级故障恢复策略。

Execution Plan

从原型走向产品的四阶段路线

阶段 1：统一事实源 把 session、turn、tool、approval、artifact、rollback 统一到一套事件协议中，结束“Python 一套、Desktop 一套”的状态。
阶段 2：统一执行引擎 选择一个权威 runtime 作为核心执行面，其他入口只做 adapter。理想方向是 Desktop、CLI、Telegram 共享同一套 session lifecycle。
阶段 3：补齐恢复和验证闭环 做 context compaction、resume、fork、replay、artifact provenance、rollback integrity、自动验证与失败恢复。
阶段 4：产品化打磨与发布 完成多语言索引、稳定模型配置、打包发布、用户文档、CI/CD、遥测和问题诊断链路。
Near-Term Priorities

下一阶段最应该先做的事

优先级最高

  • 定义统一 SessionEvent / Turn 协议
  • 决定 Python core 与 Desktop runtime 的主从关系
  • 用同一条 trace / artifact / approval 生命周期贯穿所有入口
  • 把 TypeScript / Rust 代码也纳入一致的索引和定位方案

可以稍后做

  • 更复杂的多 agent 协作
  • 更华丽的 UI 动效
  • 更多聊天表面或社交入口
  • Live2D 等非核心依赖

Definition Of Done

什么时候才算“真正可用”

工程维度

  • 统一 session lifecycle
  • 统一 trace / artifact / rollback 协议
  • 多语言 repo understanding 基本可用
  • 自动验证、失败恢复和 replay 可落地

产品维度

  • 新用户能稳定完成导入仓库、创建会话、发起任务、审查 patch
  • 失败时能明确看见原因和可恢复路径
  • 审批与权限模型可解释、可记忆、可撤销
  • 文档、部署、更新和问题诊断都面向真实用户而不是作者本人
