用户路径优先

用“可观察的工作流”来理解 agent 重点不是聊天文本，而是 run 的证据链与回放能力。

本项目的价值点在于：把工具调用、文件变更、审批、回滚等动作汇总成可视化工作流， 让用户能监督模型在仓库里到底做了什么。

最重要的对象

run：一次任务执行的统一归属单位。

tool_call/tool_result：每次工具调用的输入输出与事件。

diff/artifact/rollback：所有 mutation 必须可审查与可恢复。

推荐路径

如何体验当前仓库的能力

  1. 1) 先跑通 Python core 的最小闭环 确保 workspace 限界、patch 应用、sqlite 事件落库能工作。
  2. 2) 生成索引产物（manifest/symbols/repo_map） 让系统具备“导航真实仓库”的基本视野。
  3. 3) 启动桌面端，观察 workflow/trace 确认 UI 能消费 canonical 事件，并展示 tool/diff/artifact。
  4. 4) 体验 Skills 快速引用与并行 Agent 聊天区域闪电图标可快速引用技能；设置页可配置并行执行参数。
核心能力

已实现的关键特性

智能上下文压缩

滑动窗口 + 重要性采样 + 结构化摘要，保持 95%+ 信息保留率，节省 35% tokens。

并行 Agent 系统

LLM 驱动的任务分解 + 拓扑排序 + 真并行执行，2-3x 速度提升。

Skills 按需加载

支持 ShareAI 规范（SKILL.md + XML 包装），避免 system prompt 膨胀。

权限与风险

不要把”能做”与”允许做”混在一起

能力 含义 典型工具
read 只读仓库与索引产物，不产生 mutation。 fs_read_range / search_rg / index_query_repo_map
write 允许写文件与应用 patch，需要严格证据链与回滚语义。 patch_apply_unified_diff / fs_replace_text
exec 允许执行命令，风险最高，后续需要 approvals/sandbox 收口。 cmd_run

当前已具备 capability gating 与风险标记，但 approvals/sandbox 仍是 P3 收尾重点。

关键缺口（与深度调研对齐）

哪些地方决定”能不能进生产”

Search-Replace 编辑

需要实现类似 Aider 的 search-replace 模式，提升大文件编辑精度。

自动测试运行

每次代码变更后自动运行测试，验证修改正确性，失败时智能回滚。

语义搜索增强

基于向量的语义搜索，提升代码理解和相关文件发现能力。
