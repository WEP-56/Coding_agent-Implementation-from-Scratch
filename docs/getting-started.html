Tutorial

先把仓库跑起来，再判断下一步该改哪里

这份文档把 CodingGirl 当前可运行的部分拆成两条路线：Python Core 和 Desktop Prototype。 你不需要一次把所有东西都跑起来，更不应该把“前端页面能打开”误判成“整个 agent 已经产品可用”。

运行 Python Core 运行桌面端原型
推荐顺序

先跑 Python 测试与 CLI，再启动桌面端。这样最容易看清哪些能力是真实存在、哪些只是 UI 壳。

边界提醒

tmp/ 中的上游源码只是参考资料。你可以阅读它们来理解 Codex / OpenCode 的思路， 但当前项目的运行入口并不依赖它们。

Repository Shape

仓库结构先看懂

路径 角色 你现在应该怎么理解它
codinggirl/ Python agent core 当前最接近“真正 runtime”的部分，包含 contracts、索引、patch、SQLite、CLI。
tests/ 单元测试 验证 workspace、storage、patch、indexer、orchestrator、llm adapter、policy 等基础能力。
apps/desktop/ 桌面端原型 有真实本地交互能力，但运行时仍偏实验性质，尚未和 Python core 统一。
docs/ 新文档主入口 后续面对用户、协作者和 GitHub Pages 的事实基线。
tmp/ 上游参考 只读参考区，不是本项目的工作代码。
Path A

Python Core 快速运行

环境要求

  • Python 3.11 或更高版本
  • Windows 下可直接用 py 命令
  • 建议先在仓库根目录执行，不要在 tmp/ 中试跑

第一轮推荐动作

  • 安装开发依赖
  • 执行测试，确认基础能力没有坏
  • 初始化 SQLite
  • 跑索引与编排样例
py -m pip install -e .[dev]
py -m pytest
py -m codinggirl init
py -m codinggirl index --repo . --index-dir .codinggirl/index --max-lines 160 --focus-terms auth,login
py -m codinggirl orchestrate --repo . --db .codinggirl/codinggirl.sqlite3 --goal "replace [old] with [new] in [README.md]"

每条命令在做什么

py -m pytest 确认 workspace sandbox、patch engine、storage、indexer、orchestrator 和适配器层没有明显回归。
codinggirl init 在默认路径 .codinggirl/codinggirl.sqlite3 初始化 SQLite schema。
codinggirl index 生成 manifest.json、symbols.sqlite 和 repo_map.txt，帮助 agent 理解仓库结构。
codinggirl orchestrate 运行当前最小 orchestrator，但注意它目前只支持一种非常窄的目标语法： replace [old] with [new] in [path]。
Core Features

你可以继续试的几个入口

LLM Adapter 探针

先用 mock provider 验证协议，再切换到 OpenAI-compatible 服务。

py -m codinggirl llm-probe --provider mock --model mock-1 --prompt "hello adapter"
py -m codinggirl llm-probe --provider mock --model mock-1 --with-tool --prompt "CALL_TOOL: ping"

Telegram 本地模拟

它不是联网 bot，而是一个本地模拟入口，用来验证会话指令与状态管理。

py -m codinggirl telegram-simulate --chat-id 7 --text start
py -m codinggirl telegram-simulate --chat-id 7 --text "set_repo ."
py -m codinggirl telegram-simulate --chat-id 7 --text "goal replace [old] with [new] in [README.md]"
Path B

Desktop Prototype 运行方式

环境要求

  • Node.js 20+ 更稳妥
  • 桌面打包需要 Rust toolchain 和 Tauri 所需的系统依赖
  • 仅运行 Web 开发界面时，部分本地能力会退化到 mock / browser 受限行为

启动命令

cd apps/desktop
npm install
npm run dev

如果要直接运行 Tauri 桌面模式：

npm run tauri build

桌面端当前能体验什么

  • 仓库与会话列表
  • 三栏工作台与聊天区
  • Plan / Build / Auto 模式切换
  • 时间线、Diff、Tool Calls、Logs、Artifacts、Trace 导出
  • 审批队列、Session Permission、Memory Blocks

但要注意：桌面端的模型 tool loop 目前是 Rust 命令层内的一套独立实现，并没有直接复用 Python core 的 orchestrator。

Working Modes

Plan / Build / Auto 应该怎么用

Plan

只允许分析，不允许写入。适合先读代码、先生成计划、先观察模型是否真的理解问题。

Build

默认推荐模式。允许模型准备修改，但 mutating tools 需要显式审批，比较接近当前最安全的协作方式。

Auto

允许自动应用改动。当前只能在你已经明确知道仓库和工具行为可靠时再使用，不建议把它当成默认工作流。

Model Setup

OpenAI-compatible 配置方式

Python Core

py -m codinggirl llm-probe \
  --provider openai-compatible \
  --model gpt-4o-mini \
  --base-url https://api.openai.com \
  --api-key <KEY> \
  --prompt "hi"

Desktop Prototype

在设置页填写 Provider、Base URL、API Key 和 Model。当前桌面端会优先走 OpenAI-compatible chat completions，并通过工具循环尝试读取仓库和申请变更。

如果没有有效配置，桌面端会退回 mock 提示文本，而不是变成一个真正可修改代码的 agent。

GitHub Pages

文档站点发布

为什么这套 docs 可以直接发布

站点由 index.html、多个文档页面和一份公共 CSS 组成，不依赖构建工具。只要 GitHub 能提供静态文件托管，它就可以工作。

推荐步骤

  1. 把仓库推送到 GitHub。
  2. 打开仓库 Settings → Pages。
  3. Source 选择部署分支，目录选择 /docs。
  4. 确认站点首页使用 docs/index.html。
Troubleshooting

常见误区与排查建议

“桌面 UI 跑起来了，为什么 agent 还不够智能？”

因为 UI 能运行不代表 runtime 已统一。当前真正的核心差距在于事件模型、上下文生命周期、 工具验证与恢复语义，而不是页面能不能打开。

“orchestrate 命令为什么看起来很弱？”

这是预期内的。当前 Python orchestrator 只是一个非常窄的 MVP，用来证明 contracts、workspace、 patch、review、verify 这条链路成立，而不是完整智能编排器。
