# 深度调研-实用性跃迁

更新时间：2026-03-09

## 0. 执行摘要（结论先行）

当前项目已经具备“可演示”的主干能力：有 canonical turn/item 雏形、审批、artifact/rollback、trace/workflow UI；但要进入生产，核心阻塞不在“再加功能”，而在 **单一事实源收敛、上下文生命周期闭环、验证与发布工程化、用户可解释性** 四个方面。

对标 Codex / OpenCode / Gemini CLI 后，建议以 **90 天三阶段**推进：

- **P0（0-2周）**：先“跑稳”——统一事件源、消除轮询关键路径、建立失败可恢复边界。
- **P1（3-6周）**：再“跑聪明”——补上下文引擎（预算/压缩/剪枝/注入）与可视化解释。
- **P2（7-12周）**：最后“可生产”——工程化门禁、观测与SLO、发布与回放能力。

## 4. 90天落地路线（可执行版）

## P0（0-2周）：稳定性收敛（先跑稳）

目标：把“状态漂移/轮询/隐式语义”降到可控范围。

### 交付物

1. **Canonical Event Source v1**
   - timeline/workflow/trace 统一由 `SessionTurn.items` 投影。
2. **审批链路事件化**
   - approval pending/approved/rejected 全量进入统一 event stream。
3. **轮询降级方案**
   - 以 push 为主、poll 仅作为 fallback。
4. **失败语义统一**
   - transport/provider/model/tool/approval/rollback 对应 retry/fallback/user-facing message。

### 验收KPI

- 会话内 UI 状态不一致率 < 1%
- 审批响应可见延迟 P95 < 800ms
- 关键失败类型分类覆盖率 100%
- “执行后无归属 runId 的 mutation”= 0

## P1（3-6周）：上下文与修改能力跃迁（再跑聪明）

目标：把“可显示的上下文”变成“可执行且可验证的上下文引擎”。

### 交付物

1. **Context Engine v2 真正落地**
   - normalization、budget tracking、auto/manual compaction、tool-output prune。
2. **Context 生命周期可观测**
   - 每轮输出 token breakdown + compaction decision + before/after。
3. **修改能力安全增强**
   - read-before-write guard、重复工具调用抑制、无进展熔断。
4. **回滚一致性**
   - revert/rollback 与 compaction 语义互斥冲突归零（参考 OpenCode revert-compact 测试策略）。

### 验收KPI

- 长会话成功率（>30轮）提升到 ≥ 85%
- “上下文崩塌导致无关输出”投诉下降 50%+
- 回滚成功率 ≥ 99%
- 回滚后可继续执行成功率 ≥ 95%

## P2（7-12周）：生产化门禁与运营能力（可生产）

目标：形成“可持续发布、可追踪、可审计”的工程系统。

### 交付物

1. **CI/CD 质量门禁**
   - Python tests + Desktop build + Tauri build/test + 关键 E2E。
2. **观测与SLO**
   - turn latency、tool success、approval latency、rollback success、session resume success。
3. **Trace Bundle Replay**
   - 支持导出、回放、根因定位。
4. **分层配置系统**
   - user/project/system/env/cli 覆盖优先级，策略可控。

### 验收KPI

- 发布后 P0/P1 故障：0
- 回归漏检率下降到 < 5%
- 线上会话中断恢复成功率 ≥ 98%
- 用户可解释性评分（“我知道系统在做什么”）≥ 4.3/5

## 6. 不可妥协底线（否则无法生产）

1. **禁止双轨状态长期并存**：必须收敛到单一事实源。
2. **禁止“模型说完成即完成”**：必须有验证阶段（lint/test/build/diagnostics）与结果事件。
3. **禁止回滚旁路**：rollback 必须是 turn graph 一等事件。
4. **禁止关键路径纯轮询**：实时流是主路径，轮询是降级路径。

## 9. Oracle架构复核补充（已合并）

Oracle 复核结论与本报告主线一致：**先做状态与事件单一事实源，再做上下文生命周期，再做工程化运营能力**。

### 9.1 Oracle强调的三条阻断项（必须优先）

1. **轮询→流式事件（SSE/WebSocket）**
   - 关键面板（approval/trace/workflow）不能长期依赖 polling。
2. **禁止双轨状态**
   - UI 必须由 canonical event reducer 驱动，支持事件重放恢复。
3. **验证闭环默认开启**
   - 每个 run 必须有 verify 证据（日志/检查项/结果），失败必须可回滚。

### 9.2 Oracle建议的阶段校准（纳入本路线）

- **P0**：补事件流 + 单一事实源 + verify 证据链。
- **P1**：Context v2（compaction阈值、原因解释、checkpoint/restore、revert+compact一致性测试）。
- **P2**：安全策略分层（sandbox/approval/network/redaction/folder trust）+ 运营级可观测（失败TopN、延迟分解、回滚原因）。

### 9.3 Oracle补充KPI（并入验收）

- 事件重放一致性 ≥ 99%
- 审批事件到UI延迟 p95 ≤ 500ms
- 长会话（>30轮）成功率 ≥ 85%（P2目标）
- 失败“可行动解释”覆盖率 ≥ 95%
